3:T553,<h1 id="authentication-policy">Authentication Policy</h1><p>This project is intentionally not open signup software.</p>
<h2 id="default-sign-in-rule">Default sign-in rule</h2><p>By default, sign-in is limited to Google accounts in the domain configured by <code>ALLOWED_DOMAIN</code>. That is the default team login path.</p>
<h2 id="invited-external-users">Invited external users</h2><p>People outside the configured <code>ALLOWED_DOMAIN</code> are not allowed to sign in unless an admin has created a valid invite for their exact email address.</p>
<p>The current auth flow checks for:</p>
<ul>
<li>a matching email</li>
<li>an unused invite</li>
<li>an invite that has not expired</li>
</ul>
<p>If no valid invite exists, sign-in is rejected.</p>
<h2 id="bootstrap-admin">Bootstrap admin</h2><p>The bootstrap admin is controlled by <code>ADMIN_EMAIL</code>. The checked-in example value is <code>john@example.com</code>.</p>
<h2 id="banned-users">Banned users</h2><p>If a user is marked as banned in the database, sign-in is blocked even if they match the domain rule or were previously invited.</p>
<h2 id="why-this-is-strict">Why this is strict</h2><p>The application is built for a real team, so access control is treated as part of the product, not a later add-on. Public visitors can browse the project and docs, but member access remains restricted.</p>
