3:Tf45,<h1 id="architecture">Architecture</h1><p>Pi-Chat is a Next.js 16 application using the App Router. It runs a standalone Node server backed by PostgreSQL and MinIO, with NextAuth handling Google OAuth.</p>
<h2 id="runtime-services">Runtime services</h2><p>A fully running Pi-Chat deployment needs four services:</p>
<table>
<thead>
<tr>
<th>Service</th>
<th>Role</th>
</tr>
</thead>
<tbody><tr>
<td>Next.js standalone server</td>
<td>Renders the UI and serves API routes</td>
</tr>
<tr>
<td>PostgreSQL</td>
<td>Primary database for all user, channel, and message data</td>
</tr>
<tr>
<td>MinIO (or S3)</td>
<td>Object storage for file attachments</td>
</tr>
<tr>
<td>(Phase 3) Socket.io server</td>
<td>Realtime event delivery for messaging</td>
</tr>
</tbody></table>
<p>In the current phase, Socket.io is not yet running. The app works without it, but messages are not pushed in real time.</p>
<h2 id="application-layers">Application layers</h2><h3 id="routing">Routing</h3><p>The App Router is split into route groups:</p>
<ul>
<li><code>(public)</code>: login page, public website, and docs. No session required.</li>
<li><code>(app)</code>: the authenticated workspace shell, channels, DMs, settings.</li>
<li><code>(auth)</code>: sign-in and OAuth callback handling.</li>
<li><code>admin</code>: admin dashboard (planned, Phase 5).</li>
</ul>
<h3 id="api-routes">API routes</h3><p>All API routes live under <code>app/api/</code>. Key endpoints:</p>
<ul>
<li><code>/api/auth/[...nextauth]</code>: NextAuth handler</li>
<li><code>/api/health</code>: checks Postgres and MinIO connectivity</li>
<li><code>/api/integrations/webhooks</code>: CRUD for incoming webhooks</li>
<li><code>/api/hooks/[token]</code>: public endpoint that accepts webhook deliveries from external apps</li>
</ul>
<h3 id="data-layer">Data layer</h3><p>Prisma connects to PostgreSQL. The schema is defined in <code>prisma/schema.prisma</code> and versioned through migration files. The Prisma client is a singleton in <code>lib/prisma.ts</code>.</p>
<h3 id="storage-layer">Storage layer</h3><p>File attachments are stored in a MinIO bucket. The storage client is initialized in <code>lib/minio.ts</code>. The production Docker entrypoint creates required buckets at startup via <code>scripts/init-minio.mjs</code>.</p>
<h3 id="auth-layer">Auth layer</h3><p>NextAuth v5 handles sign-in through <code>lib/auth.ts</code>. The sign-in callback enforces the domain restriction and invite check before creating a session. See the Authentication doc for the full policy.</p>
<h3 id="state-management">State management</h3><p>Client-side state is handled with Zustand. Store slices cover channel membership, active channel context, and UI state.</p>
<h2 id="request-lifecycle">Request lifecycle</h2><ol>
<li>Browser makes a request.</li>
<li><code>proxy.ts</code> middleware checks whether the route is public or requires a session.</li>
<li>Public routes serve immediately. Protected routes redirect unauthenticated users to <code>/login</code>.</li>
<li>Server Components fetch data from Prisma and pass it to the UI.</li>
<li>Client Components handle interactions and update Zustand state.</li>
<li>Mutations go through API routes, which enforce auth and permission checks server-side.</li>
</ol>
<h2 id="production-startup-order">Production startup order</h2><p>The Docker entrypoint runs these steps before starting Next.js:</p>
<ol>
<li>Apply all pending Prisma migrations</li>
<li>Create MinIO buckets if they do not exist</li>
<li>Run idempotent bootstrap seed (admin account, default channel)</li>
<li>Start the Next.js standalone server</li>
</ol>
<h2 id="planned-realtime-server">Planned: realtime server</h2><p>Phase 3 will introduce a custom Node server that mounts both Next.js and Socket.io on the same port. The Socket.io layer will push new messages, reactions, and presence updates to connected clients without polling.</p>
