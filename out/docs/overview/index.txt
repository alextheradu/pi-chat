11:T881,<h1 id="project-overview">Project Overview</h1><p>Pi-Chat is an open-source, self-hosted team communication platform. It is a Slack alternative built for teams that want full control over where their data lives and who can access it.</p>
<h2 id="why-pi-chat-exists">Why Pi-Chat exists</h2><p>Most team chat tools are SaaS. Your messages, files, and history live on someone else&#39;s servers under someone else&#39;s terms. Pi-Chat is built on the premise that a team should be able to run their own workspace, on their own hardware, with no external dependencies at runtime.</p>
<h2 id="what-pi-chat-ships-today">What Pi-Chat ships today</h2><ul>
<li>Google OAuth sign-in with domain restriction and invite-based exceptions</li>
<li>Channel-based team messaging</li>
<li>Direct messages and group DMs</li>
<li>Incoming webhooks for external application and bot integrations</li>
<li>Role-based access control with admin, moderator, member, and guest roles</li>
<li>Task boards</li>
<li>File attachment support via MinIO-compatible object storage</li>
<li>Docker-based production deployment</li>
<li>Health checks for database and storage</li>
<li>PWA support</li>
</ul>
<h2 id="what-is-still-in-progress">What is still in progress</h2><ul>
<li>Real-time messaging via a custom Socket.io server (Phase 3)</li>
<li>Threads and global search (Phase 4)</li>
<li>Admin dashboard and member management UI (Phase 5)</li>
<li>Push notifications (Phase 6)</li>
<li>Native iOS and Android apps via Capacitor (Phase 7)</li>
</ul>
<h2 id="who-can-access-a-pi-chat-workspace">Who can access a Pi-Chat workspace</h2><p>Access is not open by default. Sign-in is restricted to Google accounts in a configured domain. People outside that domain need an admin-issued invite. This is by design: Pi-Chat is built for a real team, not a public forum.</p>
<h2 id="tech-stack">Tech stack</h2><ul>
<li>Next.js 16 with the App Router</li>
<li>Prisma 7 with PostgreSQL</li>
<li>NextAuth v5 with Google OAuth</li>
<li>MinIO-compatible object storage</li>
<li>Zustand for client-side state</li>
<li>Docker for production deployment</li>
<li>Socket.io planned for the realtime server (Phase 3)</li>
</ul>
